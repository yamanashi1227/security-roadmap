P1

フェーズ1：可視化・棚卸し（2026年）

「今クラウド上に何があり、どこが危ないか」を把握することが最優先。設定ミス・野良リソース・過剰権限は今すぐ存在しているリスク。まず見えるようにする。

設定ミスの可視化IAMの整理暗号化設定の確認
① クラウドIAM・アクセス管理
IAMロール・ポリシーの全棚卸し最優先
誰が（人・サービス・Lambda・EC2）何の権限を持っているかを一覧化。AdministratorAccess等の過剰権限を特定する
未使用のIAMユーザー・アクセスキーの特定
IAM Credentials Reportを使い90日以上未使用のキー・ユーザーを特定。退職者・旧サービスのキーが残存していないかを確認
サービスアカウントの権限確認
アプリケーションが使うサービスアカウント（SA）の権限スコープを確認。必要以上の権限を持つSAを洗い出す
② インフラ構成・ネットワーク
CSPMツールの導入と初回スキャン最優先
AWS Security Hub・Prisma Cloud・Wiz等を導入してクラウド全体の設定ミスを自動検出。公開S3バケット・過剰なSG・MFA無効などを一括把握
外部公開リソースの棚卸し
パブリックIPが付いているEC2・RDS・S3・ELBをリスト化。意図せず公開されているリソースを即時クローズ
セキュリティグループ・NSGの確認
0.0.0.0/0へのInboundを許可しているルールを特定。SSH・RDPの全公開は最高リスクとして即時修正対象
③ コード・リポジトリ
ソースコードへの秘密情報埋め込みの調査最優先
git-secrets・truffleHog・GitGuardianで既存リポジトリをスキャン。APIキー・DB接続文字列・証明書がコードに含まれていないかを確認
IaC（Terraform/CDK）のセキュリティ設定確認
tfsec・checkovで既存コードをスキャン。暗号化未設定・パブリックアクセス許可・ログ無効などの設定ミスを特定
コンテナイメージの脆弱性スキャン（初回）
Trivy・Snyk・ECR Scanningで現在使用中のコンテナイメージをスキャン。Critical/High脆弱性の件数と修正優先度を把握
④ 監視・ログの現状確認
CloudTrail / 監査ログの有効化確認
AWS CloudTrail・GCP Cloud Audit Logs等の管理操作ログが有効かを確認。全リージョンで有効でない場合は即時有効化
VPCフローログの有効化確認
不審な通信パターンを検知するためにVPCフローログを有効化。保存先（S3・CloudWatch）とリテンション期間を設定
アラート設定の現状把握
ルートアカウントログイン・IAM変更・SG変更等にアラートが設定されているかを確認し、空白を特定
⑤ 暗号化：クラウドリソースの暗号化設定棚卸し暗号化 Phase 1
S3バケット暗号化の確認最優先
全S3バケットのSSE設定を確認。SSE-S3・SSE-KMS・SSE-Cの区別と、バケットポリシーで非暗号化アップロードを拒否しているかを確認
RDS・DBの暗号化状態確認
RDS・DynamoDB・ElastiCache等のストレージ暗号化が有効かを確認。暗号化は作成時のみ有効化可能なため、未暗号化DBは移行計画を策定
EBS・ディスクボリュームの暗号化確認
EC2にアタッチされているEBSボリュームの暗号化状態を確認。アカウントレベルでEBSデフォルト暗号化が有効になっているかを確認
KMSキーの棚卸し
現在作成されているKMSキーの一覧化。AWS管理キー vs CMKの区別、キーポリシー・使用状況・ローテーション設定を確認
TLS証明書・ACMの棚卸し
ACM等で管理している証明書の有効期限・ドメイン・使用サービスを一覧化。90日以内に期限切れになる証明書を特定
シークレット管理の現状確認
DB接続文字列・APIキー・OAuthシークレット等がどこに保存されているかを確認。環境変数・設定ファイル・Secrets Managerの使用状況を把握
CSPMスキャン
完了
公開S3バケット
ゼロへ
コード内クレデンシャル
全件把握
このフェーズで最も多く見つかるリスク：①コードに埋め込まれたAWSアクセスキー ②誰でも読めるS3バケット ③0.0.0.0/0を許可したSG ④使われていない管理者IAMユーザー ⑤暗号化されていないRDS。これらは「設定を変えるだけ」で今日から対処できる低コスト・高効果の対策。

フェーズ2：基盤堅牢化（2027年）

棚卸しで見つかった問題を体系的に修正し、「セキュアなデフォルト」を確立する。IAMは最小権限を原則化し、シークレットはコードから追い出す。暗号化はKMSによる鍵管理の一元化が目標。

最小権限の原則化ネットワーク分離KMS・Secrets Manager導入
① IAMの最小権限化
IAMポリシーの最小権限化（Least Privilege）最優先
IAM Access Analyzerで実際に使われていないアクションを特定して権限を絞り込む。「*」（ワイルドカード）アクション・リソースを排除
ルートアカウントのMFA強制と封印
AWSルートアカウントはMFAを設定した上で日常使用を禁止。アクセスキーは作成・使用しない。ルートログインをCloudTrailでアラート化
長期アクセスキーの廃止・IAMロールへの移行
EC2・Lambda・ECSから長期アクセスキーを廃止し、IAMロールによる一時クレデンシャルへ移行
Permission Boundaryの導入
開発者がIAMロールを自由に作成できる環境では、最大権限の上限を設定するPermission Boundaryを全IAMエンティティに適用
② ネットワーク・インフラ構成の強化
VPC設計の見直し・マイクロセグメンテーション最優先
パブリック/プライベートサブネットを明確に分離。DBはプライベートサブネットのみ。サービス間通信はSGで明示的に許可
セキュリティグループの最小化
0.0.0.0/0の全廃。SSHはSystems Manager Session Managerに移行してポートを閉鎖。ALBからのみアクセス可能な構成へ
WAFの導入
AWS WAF・CloudFront等でSQLインジェクション・XSS・DDoSからAPIとWebアプリを保護。マネージドルールセットを有効化
AWS Config Rulesによる自動コンプライアンスチェック
SGの変更・S3のパブリックアクセス・暗号化設定の変化をAWS Configで自動検知・アラート化
③ CI/CDへのセキュリティ組み込み（基礎）
Pre-commitフックでのシークレット検出
git-secrets・detect-secretsをpre-commitフックに設定。コミット前にAPIキー・パスワードの混入を自動ブロック
コンテナイメージスキャンのCI組み込み
CI/CDパイプラインにTrivyスキャンを追加。Critical脆弱性が検出された場合はデプロイをブロック
IaCスキャン（tfsec/checkov）のCI組み込み
Terraform・CDKの変更時にtfsec・checkovを自動実行。高リスクな設定ミスを検出してPRをブロック
④ 監視・アラートの整備
AWS GuardDuty / Defender for Cloudの有効化最優先
クラウドネイティブの脅威検知サービスを全リージョンで有効化。クレデンシャル漏洩・マイニング・異常なAPIコールを自動検知
重要イベントのアラート設定
ルートログイン・IAMポリシー変更・SG変更・CloudTrail無効化をCloudWatch Alarms・SNSでアラート化
ログの集中管理と保全期間の設定
CloudTrail・VPCフローログ・ALBアクセスログをS3に集約。S3 Object Lockで改ざん防止。保全期間は最低90日
⑤ 暗号化：KMS一元化・Secrets Manager導入・全ストレージ暗号化完了暗号化 Phase 2
CMK（カスタマー管理キー）の整備最優先
AWS KMSでサービス・環境（prod/stg/dev）ごとにCMKを作成。キーポリシーで「誰がキーを使えるか」を明示的に定義。AWS管理キーからCMKへの移行計画を策定
Secrets Managerへの完全移行最優先
DB接続文字列・APIキー・OAuthシークレットをAWS Secrets Manager・HashiCorp Vault・Azure Key Vaultに移行。環境変数へのハードコードを完全廃止
S3全バケットのSSE-KMS適用
全S3バケットにSSE-KMSを適用。バケットポリシーで非暗号化アップロードを拒否するポリシーを追加
RDS・DBの暗号化移行完了
未暗号化のRDSはスナップショット→暗号化スナップショット→新インスタンス作成のフローで移行。メンテナンスウィンドウで計画的に実施
シークレットの自動ローテーション設定
Secrets ManagerのRotation機能でDB接続パスワードを自動ローテーション（30〜90日）。アプリケーション側はSDKを使い動的に最新シークレットを取得
ALB・API GatewayのTLS設定強化
TLS 1.0/1.1を廃止しTLS 1.2以上を強制。ACMで証明書を自動更新。セキュリティポリシーを最新に更新
IAM長期キー廃止
100%
ストレージ暗号化
全適用
Secrets Manager移行
完了
このフェーズの最重要施策：コードからシークレットを追い出すこと（Secrets Manager移行）とIAMの最小権限化はセット。片方だけでは意味がない。GuardDutyは月数百円で有効化できるため、このフェーズの最初に有効化すべき。
P3

フェーズ3：DevSecOps確立（2028年）

「セキュリティをリリース後に追加する」から「開発段階から組み込む（Shift-Left）」への転換。CI/CDパイプラインを防御ラインとして機能させ、人の確認に頼らない自動化を確立する。

Shift-Left SecurityサプライチェーンセキュリティmTLS・BYOK・Envelope暗号化
① CI/CDパイプラインのセキュリティ強化
SAST（静的解析）のパイプライン組み込み最優先
Semgrep・SonarQube・CodeQLをCI/CDに組み込み、コードプッシュ時にSQLインジェクション・XSS・認証不備等の脆弱なコードパターンを自動検出
SCA（ソフトウェア構成分析）の導入
Dependabot・Snyk・OWASP Dependency-Checkでサードパーティライブラリの脆弱性を自動検知。Critical脆弱性で自動ブロック
コンテナイメージの署名（Cosign）
Sigstore/CosignでビルドしたコンテナイメージにデジタルSignatureを付与。Kubernetesのアドミッションコントローラーで署名なしイメージのデプロイをブロック
CI/CD環境自体のIAM権限最小化
CI/CDが使うIAMロールを最小権限化。OIDC連携で長期クレデンシャルを廃止
② コンテナ・Kubernetesセキュリティ
非rootユーザーでのコンテナ実行の徹底
全Dockerfileを見直し、rootユーザーでの実行を廃止。USER命令で専用ユーザーを指定。ReadOnlyRootFilesystemをPodSpecに設定
Kubernetes RBACの最小権限化
ClusterAdmin権限を排除。ServiceAccountは名前空間ごとに作成し、必要な操作のみ許可。定期的なRBACアクセス棚卸しを実施
Falcoによるランタイムセキュリティ導入
Falcoでコンテナ実行時の異常行動（シェル起動・不審なファイルアクセス・ネットワーク接続）を検知。SIEM・Slackへのアラート連携
③ ゼロトラスト・サービス間通信
サービスメッシュ（mTLS）の導入重要
Istio・Linkerd・AWS App Meshでサービス間通信をmTLSで暗号化。サービス間の認証・認可をコード外で一元管理。トラフィックの可視化も実現
API認証の強化（JWT・OAuth2）
内部APIもJWT/OAuth2で認証を要求。API GatewayでAPIキーによるレート制限・認証を実施。未認証エンドポイントのゼロ化
Workload Identity Federationの活用
クラウド間・CI/CD間のサービス認証をOIDC/Workload Identity Federationに移行。長期クレデンシャルを使わないサービス間認証を実現
④ IaC・構成管理のセキュリティ
IaCセキュリティテストの自動化
全インフラ変更をTerraform/CDK経由のみに制限。OPA/Regoでポリシーを定義し、違反するIaC変更をCI/CDでブロック
ドリフト検知の自動化
AWS Config・Terraform Cloudでインフラのドリフト（IaC定義との乖離）を自動検知。コンソールからの手動変更を即時検出してアラート
サードパーティTerraformモジュールの審査
外部のTerraformモジュール・コンテナベースイメージのソース・バージョンを固定。サプライチェーン攻撃への対策
⑤ 暗号化：mTLS・BYOK・Envelope Encryptionの実装暗号化 Phase 3
サービス間mTLS暗号化の実装最優先
サービスメッシュまたはTLS相互認証でマイクロサービス間の全通信を暗号化。「同じVPC内だから安全」という前提を捨て、東西トラフィックも暗号化する
BYOK（Bring Your Own Key）の本格実装
S3・RDS・EBS・DynamoDB等の主要ストレージをCMKで暗号化。SaaSベンダーのデフォルト鍵への依存をなくし、自社でキーのライフサイクルを管理
Envelope Encryptionの実装
KMSのEnvelope Encryption（データ暗号化キーをKMSで暗号化）をアプリレベルで実装。特に機密性の高いフィールドに適用。AWS Encryption SDKを活用
KMSキーポリシーの精査と自動監査
全CMKのキーポリシーを精査し、不要なアクセス権を削除。AWS Config RuleでキーポリシーのコンプライアンスチェックをCI/CDに組み込み
証明書管理の完全自動化
ACMによるTLS証明書の自動更新を全ドメインで設定。Private CAを構築し内部サービス間通信用の証明書を自動発行・更新。有効期限切れをゼロに
フィールドレベル暗号化の適用
DBに保存する個人情報・センシティブデータをアプリレベルで暗号化。DBへの不正アクセスがあっても平文データが漏洩しない設計
SAST/SCAのCI組み込み
完了
mTLS適用率
全サービス
コンソール手動変更
禁止化
このフェーズの核心：「セキュリティをレビューで担保する」から「パイプラインが通らない設計にする」への転換。開発者がミスしても自動でブロックされる仕組みが完成すれば、チームが拡大しても品質を維持できる。
P4

フェーズ4：継続監視・自動化（2029年）

個別の対策を「継続的に機能する仕組み」に昇華させる。検知から自動対応まで繋がったループを構築し、人的コストを削減しながら防御レベルを向上させる。

検知→対応の自動化CNAPP統合鍵ローテ自動化・PQC調査
① クラウドネイティブ脅威検知の強化
CNAPPの導入重要
Wiz・Prisma Cloud・Orca SecurityなどCNAPPを導入し、CSPM・CWPP・IaCスキャン・コンテナセキュリティを一元化。リスクの優先度付けを自動化
GuardDuty検出結果のSOAR連携
GuardDutyのFindingsをEventBridge経由でLambdaに連携し、自動対応（IAMキーの無効化・SG変更の自動リバート等）を実装
異常なAPIコールパターンの検知
CloudTrail Insightsを有効化し、通常と異なるAPIコールボリュームを自動検知。大量のDescribe*/List*は情報収集フェーズの攻撃者の挙動として警戒
② セキュリティ自動化の深化
セキュリティゲートウェイの自動化
CSPM・SAST・SCAスコアが閾値を超えた場合のデプロイブロックを自動化。セキュリティスコアをリリース基準の一部として組み込み
脆弱性パッチの自動適用
OS・ミドルウェアの脆弱性パッチをAWS Systems Manager Patch Managerで自動適用。コンテナベースイメージの定期自動更新パイプラインを整備
SBOMの生成・管理の自動化
ソフトウェア部品表（SBOM）をCIで自動生成・更新。新しいCVEが発表された際に影響を受けるコンポーネントを即時特定できる体制を整備
③ セキュリティ指標のKPI化
クラウドセキュリティKPIダッシュボード
CSPMスコア・Critical脆弱性数・未修正のIAMリスク数・暗号化カバレッジ率・ログ有効化率をKPIとして定義。週次・月次で経営報告
ペネトレーションテストの定期実施
外部専門業者によるクラウド環境のペネトレーションテストを年1回実施。IAMのprivilege escalation・メタデータサービス悪用等を重点検証
インシデント対応演習（クラウド特化）
「S3バケットから大量データが流出した」「IAMキーがGitHubに漏洩した」等のクラウド特有シナリオでの対応演習を年2回実施
④ コンプライアンス自動化
AWS Security Hub / Compliance Centerの活用
CIS AWS Foundations Benchmark・PCI DSS・ISO 27001等への準拠状況をSecurity Hubで自動評価。違反項目を自動チケット化
SOC2対応のための証跡自動収集
SOC2 Type2取得を見据えた証跡（アクセスログ・変更履歴・インシデント記録）の自動収集を整備。監査対応コストを大幅削減
マルチアカウント・マルチリージョン管理の一元化
AWS Organizations・Control Towerでセキュリティポリシーを全アカウントに一括適用。SCPでリージョン制限・危険なアクションを組織全体で禁止
⑤ 暗号化：鍵ローテーション完全自動化・ポスト量子暗号調査開始暗号化 Phase 4
KMSキーローテーションの完全自動化最優先
全CMKの年次自動ローテーションを有効化。古いキーで暗号化されたデータの再暗号化バッチジョブを整備。ローテーション状況をKPIとして定期モニタリング
AWS CloudHSM / HSMの導入検討
特に機密性の高いキー（ルートCA・マスターキー）の保管にHSMを活用。FIPS 140-2 Level 3が必要な場合はCloudHSM、そうでなければKMSで十分
弱い暗号化アルゴリズムの完全廃止
MD5・SHA-1・RSA-1024・DES等を全サービスから廃止。TLS 1.0/1.1をこのフェーズで完全廃止。暗号化アルゴリズムの棚卸しを年次で自動スキャン化
ポスト量子暗号（PQC）の影響調査将来対策
NISTのPQC標準（ML-KEM・ML-DSA・SLH-DSA）をベースに、自社サービスの暗号化実装への影響範囲を特定。TLS通信・APIキー・長期保存データへの影響を評価
HNDL対策の優先データ特定将来対策
Harvest Now, Decrypt Later攻撃に対して、長期保護が必要なデータ（顧客PII・契約情報・決済履歴）を特定し、PQC移行の優先順位を設定
暗号化実装の外部監査
外部の暗号化専門家による実装監査を実施。KMSキーポリシー・Secrets Managerのアクセス設定・TLS設定が適切に実装されているかを第三者視点で検証

⑤ 暗号化：ポスト量子暗号移行と暗号化アジリティの完成暗号化 Phase 5
NIST PQCアルゴリズムへの段階移行最重要
ML-KEM（鍵交換）・ML-DSA（署名）への移行を開始。AWS・GCPのPQC対応TLSが利用可能になり次第採用。優先対象は外部公開API・長期データ保護・認証基盤

全データ暗号化の最終監査

顧客向け暗号化透明性の実現
