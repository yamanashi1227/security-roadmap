フェーズ 1：可視化・棚卸し（2026 年）

「今 Azure 上に何があり、どこが危ないか」を把握することが最優先。設定ミス・野良リソース・過剰権限は今すぐ存在しているリスク。まず見えるようにする。

設定ミスの可視化RBAC の整理暗号化設定の確認
① Azure RBAC・Entra ID 管理
RBAC ロール割り当ての全棚卸し最優先
サブスクリプション・リソースグループ・リソース単位で Owner／Contributor 等の権限を一覧化。Microsoft Entra ID のアクセスレビュー機能で過剰権限を特定する
未使用のサービスプリンシパル・アプリ登録の特定
Entra ID の Sign-in Logs・アプリ登録リストで 90 日以上未使用のサービスプリンシパルを特定。退職者・旧サービスのクレデンシャルが残存していないかを確認
マネージド ID の権限確認
App Service・Azure Functions・VM が使うマネージド ID（システム割り当て／ユーザー割り当て）の RBAC スコープを確認。必要以上の権限を持つ ID を洗い出す
② インフラ構成・ネットワーク
Microsoft Defender for Cloud の有効化と初回スキャン最優先
Defender for Cloud のセキュリティスコアを確認。パブリック公開された Blob コンテナ・過剰な NSG ルール・MFA 無効などを一括把握。推奨事項のリストを優先度順に整理
外部公開リソースの棚卸し
パブリック IP が付いている VM・App Service・Azure SQL・Load Balancer・Storage Account をリスト化。意図せず公開されているリソースを即時クローズ
NSG（ネットワークセキュリティグループ）の確認
0.0.0.0/0 へのインバウンドを許可している NSG ルールを特定。RDP（3389）・SSH（22）の全公開は最高リスクとして即時修正対象
③ コード・リポジトリ
ソースコードへの秘密情報埋め込みの調査最優先
git-secrets・truffleHog・GitGuardian で既存リポジトリをスキャン。Azure 接続文字列・ストレージアクセスキー・証明書がコードに含まれていないかを確認
IaC（Bicep／Terraform）のセキュリティ設定確認
tfsec・checkov で既存コードをスキャン。暗号化未設定・パブリックアクセス許可・診断ログ無効などの設定ミスを特定
コンテナイメージの脆弱性スキャン（初回）
Trivy・Microsoft Defender for Containers・Azure Container Registry（ACR）の組み込みスキャンで使用中のコンテナイメージをスキャン。Critical／High 脆弱性の件数と修正優先度を把握
④ 監視・ログの現状確認
Azure Monitor Activity Log の有効化確認
サブスクリプション単位の管理操作ログ（Activity Log）が有効かを確認。Log Analytics Workspace への転送設定・保持期間（最低 90 日）を確認
Azure Network Watcher フローログの有効化確認
VNet フローログを有効化し不審な通信パターンを検知できる状態に。Log Analytics Workspace への転送と NSG フローログの保存設定を確認
アラート設定の現状把握
グローバル管理者へのロール昇格・NSG 変更・Azure Policy 変更等に Azure Monitor Alerts が設定されているかを確認し、空白を特定
⑤ 暗号化：Azure リソースの暗号化設定棚卸し暗号化 Phase 1
Azure Blob Storage 暗号化の確認最優先
全ストレージアカウントの SSE（Storage Service Encryption）設定を確認。Microsoft 管理キー vs カスタマー管理キー（CMK）の区別と、パブリックアクセスが無効になっているかを確認
Azure SQL・DB の暗号化状態確認
Azure SQL Database・Cosmos DB・Azure Database for PostgreSQL／MySQL 等の TDE（Transparent Data Encryption）が有効かを確認。CMK vs サービス管理キーの区別を把握
Azure Managed Disks の暗号化確認
VM にアタッチされている Managed Disks の暗号化状態を確認。Azure Disk Encryption（ADE）または サーバー側暗号化（SSE）が有効になっているかを確認
Azure Key Vault キーの棚卸し
現在作成されている Key Vault のキー・シークレット・証明書の一覧化。アクセスポリシー vs RBAC 認証の区別、使用状況・ローテーション設定・ソフト削除の有効化状態を確認
TLS 証明書・Key Vault Certificates の棚卸し
Key Vault Certificates・App Service Certificates 等で管理している証明書の有効期限・ドメイン・使用サービスを一覧化。90 日以内に期限切れになる証明書を特定
シークレット管理の現状確認
接続文字列・APIキー・OAuth シークレット等がどこに保存されているかを確認。App Service の環境変数・Key Vault References・Azure App Configuration の使用状況を把握
Defender for Cloud スキャン
完了

全件把握
このフェーズで最も多く見つかるリスク：①コードに埋め込まれたストレージアクセスキー・接続文字列 ②パブリックアクセスが有効な Blob コンテナ ③0.0.0.0/0 を許可した NSG ④使われていない Owner 権限のサービスプリンシパル ⑤暗号化されていない Azure SQL Database。これらは「設定を変えるだけ」で今日から対処できる低コスト・高効果の対策。

フェーズ 2：基盤堅牢化（2027 年）

棚卸しで見つかった問題を体系的に修正し「セキュアなデフォルト」を確立する。RBAC は最小権限を原則化し、シークレットはコードから追い出す。暗号化は Key Vault CMK による鍵管理の一元化が目標。

最小権限の原則化VNet 分離強化Key Vault CMK・Secrets 移行
① RBAC・Entra ID の最小権限化
RBAC の最小権限化（Least Privilege）最優先
Entra ID Access Reviews で実際に使われていないロール割り当てを特定。Owner／Contributor を排除し、必要な操作のみ許可するカスタムロールへ移行
Privileged Identity Management（PIM）の導入
グローバル管理者・Owner 等の特権ロールを PIM で Just-in-Time（JIT）に制限。承認ワークフロー・アクティベーション期限・監査ログを整備し、常時特権を廃止
ストレージアカウントキーの廃止・マネージド ID へ移行
App Service・Azure Functions・VM からストレージアカウントキーを廃止し、マネージド ID（Managed Identity）による認証へ移行。長期クレデンシャルをゼロに
Conditional Access Policies の設定
Entra ID の条件付きアクセスポリシーを設定。リスクの高いサインインの自動ブロック・準拠デバイスからのみアクセス許可・特定国外からのアクセス遮断
② ネットワーク・インフラ構成の強化
VNet 設計の見直し・サブネット分離最優先
パブリック／プライベートサブネットを明確に分離。Azure Private Endpoint で Azure SQL・Storage・Key Vault 等へのプライベート接続を実現。サービスエンドポイントを段階的に廃止
NSG の最小化・Azure Bastion の導入
0.0.0.0/0 の全廃。RDP／SSH は Azure Bastion に移行してポートを閉鎖。Application Gateway からのみアクセスできるよう NSG と UDR を設定
Azure WAF の導入
Azure Application Gateway WAF v2・Azure Front Door WAF で SQL インジェクション・XSS・DDoS から API と Web アプリを保護。OWASP マネージドルールセットを有効化
Azure Policy による自動コンプライアンスチェック
NSG の変更・Storage のパブリックアクセス・暗号化設定の変化を Azure Policy で自動検知・修復（DeployIfNotExists）。Built-in ポリシーイニシアチブを活用
③ CI/CD へのセキュリティ組み込み（基礎）
Pre-commit フックでのシークレット検出
git-secrets・detect-secrets を pre-commit フックに設定。コミット前に Azure 接続文字列・API キーの混入を自動ブロック。全開発者の環境に適用
コンテナイメージスキャンの CI 組み込み
Azure DevOps・GitHub Actions パイプラインに Trivy スキャンを追加。Critical 脆弱性が検出された場合はデプロイをブロック。ACR タスクによる自動スキャンも併用
IaC スキャン（tfsec／checkov）の CI 組み込み
Bicep／Terraform の変更時に tfsec・checkov を自動実行。高リスクな設定ミス（暗号化無効・パブリックアクセス許可等）を検出して PR をブロック
④ 監視・アラートの整備
Microsoft Defender for Cloud の全サブスクリプション有効化最優先
Defender for Servers・Defender for SQL・Defender for Containers・Defender for Key Vault を有効化。クレデンシャル漏洩・マルウェア・異常な API 操作を自動検知
重要イベントの Azure Monitor Alerts 設定
グローバル管理者への昇格・RBAC 変更・NSG 変更・Key Vault 削除・Defender for Cloud 無効化を Azure Monitor Alerts・アクショングループ（メール／Teams）でアラート化
Log Analytics Workspace へのログ集中管理
Activity Log・NSG フローログ・App Gateway アクセスログ・Entra ID サインインログを Log Analytics に集約。保全期間を最低 90 日に設定し、改ざん防止のための不変ストレージを構成
⑤ 暗号化：Key Vault CMK 一元化・Secrets 移行・全ストレージ暗号化完了暗号化 Phase 2
Key Vault Premium＋CMK の整備最優先
Key Vault Premium（HSM バックド）でサービス・環境（prod／stg／dev）ごとに CMK を作成。アクセスポリシーから RBAC 認証へ移行。Microsoft 管理キーから CMK への切り替え計画を策定
Key Vault Secrets への完全移行最優先
Azure SQL 接続文字列・API キー・OAuth クライアントシークレットを Key Vault Secrets に移行。App Service の「Key Vault References」機能でアプリコードを変更せずに安全参照。環境変数へのハードコードを完全廃止
Azure Blob Storage 全コンテナの CMK 適用
全ストレージアカウントに CMK（Key Vault 管理キー）を設定。Storage Account のパブリックアクセスを Deny に設定。非暗号化転送（HTTP）を拒否するポリシーを適用
Azure SQL・DB の CMK 暗号化移行完了
Azure SQL の TDE を CMK（Key Vault 管理）に移行。Transparent Data Encryption のキーを Azure Key Vault で管理し、データベース側の暗号化キーを自社管理下に置く
Key Vault Secrets の自動ローテーション設定
Key Vault のローテーションポリシーと Azure Functions を組み合わせ、DB パスワード・API キーを自動ローテーション（30〜90 日）。アプリ側は Key Vault Reference でシークレット更新を自動反映
App Gateway・API Management の TLS 設定強化
TLS 1.0／1.1 を廃止し TLS 1.2 以上を強制。App Service Certificate または Key Vault Certificates で証明書を自動更新。Azure Front Door の最小 TLS バージョンを 1.2 に設定
長期クレデンシャル廃止
100%
ストレージ暗号化
全適用
Key Vault Secrets 移行
完了
このフェーズの最重要施策：コードからシークレットを追い出すこと（Key Vault Secrets 移行）と RBAC の最小権限化はセット。Defender for Cloud は有効化するだけでセキュリティスコアが可視化され、優先対応項目が自動的に提示される。このフェーズの最初に全サブスクリプションで有効化すべき。

フェーズ 3：DevSecOps 確立（2028 年）

「セキュリティをリリース後に追加する」から「開発段階から組み込む（Shift-Left）」への転換。CI/CD パイプラインを防御ラインとして機能させ、人の確認に頼らない自動化を確立する。

Shift-Left SecurityサプライチェーンセキュリティmTLS・BYOK・Always Encrypted
① CI/CD パイプラインのセキュリティ強化
SAST（静的解析）のパイプライン組み込み最優先
Semgrep・SonarQube・GitHub Advanced Security（CodeQL）を Azure DevOps／GitHub Actions に組み込み。SQL インジェクション・XSS・認証不備等の脆弱なコードパターンをプッシュ時に自動検出
SCA（ソフトウェア構成分析）の導入
Dependabot・Snyk・OWASP Dependency-Check でサードパーティライブラリの脆弱性を自動検知。Azure DevOps の Dependency Scanning または GitHub Advanced Security を活用
コンテナイメージの署名（Notation／Cosign with ACR）
Azure Container Registry（ACR）と Notation（CNCF）でビルドしたコンテナイメージにデジタル署名を付与。AKS の Azure Policy アドオンで署名なしイメージのデプロイをブロック
CI/CD 環境自体の権限最小化（OIDC 連携）
Azure DevOps Service Connection・GitHub Actions の Workload Identity Federation（OIDC）で長期クレデンシャルを廃止。「デプロイに必要な権限のみ」の RBAC に絞り込み
② コンテナ・AKS セキュリティ
非 root ユーザーでのコンテナ実行の徹底
全 Dockerfile を見直し root ユーザーでの実行を廃止。USER 命令で専用ユーザーを指定。AKS の Pod Security Admission で Restricted プロファイルを適用
AKS RBAC の最小権限化（Azure RBAC for Kubernetes）
AKS に Azure RBAC 統合を有効化し Entra ID でアクセス制御を一元管理。ClusterAdmin 権限を排除。Namespace 単位の ServiceAccount は必要な操作のみ許可
Microsoft Defender for Containers によるランタイムセキュリティ
Defender for Containers を有効化しコンテナ実行時の異常行動（シェル起動・不審なファイルアクセス・ネットワーク接続）を検知。Microsoft Sentinel・Teams へのアラート連携
③ ゼロトラスト・サービス間通信
AKS でのサービスメッシュ（mTLS）導入重要
AKS の Open Service Mesh アドオン・Istio ベースのサービスメッシュでサービス間通信を mTLS で暗号化。サービス間の認証・認可をコード外で一元管理しトラフィックを可視化
Azure API Management での OAuth2／JWT 認証強化
Azure API Management（APIM）で全 API に OAuth2／JWT 認証を要求。Entra ID を Identity Provider として設定。未認証エンドポイントをゼロ化
Azure Workload Identity によるサービス間認証
AKS から Azure リソースへのアクセスに Azure Workload Identity（OIDC）を採用。Pod に割り当てたマネージド ID でサービス認証を実現し長期クレデンシャルを完全廃止
④ IaC・構成管理のセキュリティ
Bicep／Terraform IaC のセキュリティテスト自動化
全インフラ変更を Bicep／Terraform 経由のみに制限（ポータル直接変更を禁止）。Azure Policy as Code・OPA／Rego でポリシーを定義し違反する IaC 変更を CI/CD でブロック
ドリフト検知の自動化
Azure Policy・Terraform Cloud でインフラのドリフト（IaC 定義との乖離）を自動検知。Azure ポータルからの手動変更を即時検出してアラート。重大変更は自動リバートを検討
サードパーティ Bicep／Terraform モジュールの審査
外部の Terraform モジュール・コンテナベースイメージのソース・バージョンを固定。Bicep Registry（プライベート）への移行でサプライチェーン攻撃リスクを低減
⑤ 暗号化：mTLS・BYOK・Azure SQL Always Encrypted の実装暗号化 Phase 3
サービス間 mTLS 暗号化の実装最優先
AKS サービスメッシュで東西トラフィック（サービス間）を mTLS で暗号化。「同じ VNet 内だから安全」という前提を捨て、サービス間の全通信を暗号化・認証する
BYOK（Bring Your Own Key）の本格実装
Azure Storage・Azure SQL・Azure Disk・Azure Monitor の暗号化キーを全て CMK（Azure Key Vault 管理）に移行。Microsoft 管理キーへの依存をなくし、自社でキーのライフサイクルを管理
Azure SQL Always Encrypted の適用
個人情報・決済情報等の機密フィールドに Azure SQL Always Encrypted を適用。データベースサーバーや管理者からも平文が見えない列レベル暗号化を実現。暗号化キーは Key Vault で管理
Key Vault キーポリシーの精査と自動監査
全 CMK のアクセスポリシー（または RBAC）を精査し、不要なアクセス権を削除。Azure Policy Rule で Key Vault のポリシー変更をリアルタイム検知し、変更を Sentinel に転送
Key Vault Certificates による証明書管理の完全自動化
Key Vault Certificates で TLS 証明書を自動更新（Let's Encrypt・DigiCert との統合）。App Service・App Gateway・APIM への証明書自動配布で有効期限切れをゼロに
Microsoft Purview によるデータ分類とフィールドレベル暗号化
Microsoft Purview でデータ資産を自動分類（機密・社外秘・一般）。分類結果に基づき機密データに対してフィールドレベル暗号化を適用。Azure SQL Dynamic Data Masking と組み合わせて保護
SAST/SCA の CI 組み込み
完了
mTLS 適用率
全サービス
ポータル手動変更
禁止化
このフェーズの核心：「セキュリティをレビューで担保する」から「パイプラインが通らない設計にする」への転換。Azure DevOps／GitHub Actions でのセキュリティゲートが完成すれば、チームが拡大しても品質を維持できる。

フェーズ 4：継続監視・自動化（2029 年）

個別の対策を「継続的に機能する仕組み」に昇華させる。Microsoft Sentinel を中心に検知から自動対応まで繋がったループを構築し、人的コストを削減しながら防御レベルを向上させる。

Sentinel SIEM/SOARDefender CNAPP 統合鍵ローテ自動化・PQC 調査
① Microsoft Sentinel による脅威検知強化
Microsoft Sentinel の導入（SIEM/SOAR）重要
Microsoft Sentinel を導入し Log Analytics Workspace 上に SIEM を構築。Defender for Cloud・Entra ID・Activity Log・NSG フローログ等のデータコネクタを統合。ML ベースの異常検知を有効化
Sentinel Automation Rules・Playbook による自動対応
Sentinel の Automation Rules と Logic Apps（Playbook）で検知後の自動対応を実装。Entra ID アカウントの自動無効化・NSG ルールの自動リバート・Teams チャンネルへの自動通知
異常な管理操作パターンの検知
Entra ID Identity Protection・Sentinel Analytics Rules で通常と異なる API 操作・サインインパターンを自動検知。大量の LIST・GET 操作は情報収集フェーズの攻撃者の挙動として警戒
② セキュリティ自動化の深化
Defender for DevOps によるセキュリティゲートウェイ自動化
Microsoft Defender for DevOps を Azure DevOps／GitHub と統合。セキュリティスコアが閾値を超えた場合のデプロイブロックを自動化。PR への自動セキュリティコメントを実装
Azure Update Manager による脆弱性パッチの自動適用
Azure Update Manager で VM・VMSS の OS パッチを自動適用。メンテナンス構成でビジネス時間外の自動更新をスケジュール。AKS ノードプールの自動アップグレードも有効化
SBOM の生成・管理の自動化
ソフトウェア部品表（SBOM）を CI で自動生成・Azure Container Registry に保存。新しい CVE が発表された際に影響を受けるコンポーネントを即時特定できる体制を整備
③ セキュリティ指標の KPI 化
Defender for Cloud セキュリティスコアの KPI ダッシュボード
